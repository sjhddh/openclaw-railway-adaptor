## What this is

A thin wrapper repo. Click **Deploy on Railway** → Railway pulls
[`ghcr.io/openclaw/openclaw`](https://github.com/openclaw/openclaw/pkgs/container/openclaw)
(the upstream prebuilt image), wraps it with Railway-friendly entrypoint and
config, and runs your own openclaw gateway behind a public URL.

A daily GitHub Actions workflow pins the wrapper to the latest upstream digest,
smoke-tests it, and pushes — which triggers Railway to redeploy. **You don't
maintain the build.** Upstream's improvements and security patches arrive
automatically, the morning after they're released.

### What deploys

OpenClaw is a self-hosted multi-channel personal AI assistant. The Railway
service runs the **gateway** — an HTTP/WebSocket server that bridges chat
channels (Telegram, Discord, Slack, Matrix, iMessage, Signal, IRC, Feishu,
LINE, …) to LLM providers (Anthropic, OpenAI, Gemini, OpenRouter). You
configure channels and keys via the gateway UI/API after deploy.

### What this wrapper does

| Concern | How it's handled |
| --- | --- |
| **Bind 0.0.0.0** | `start.sh` execs the gateway with `--bind lan` so Railway's proxy can reach it. |
| **Persistent state** | `OPENCLAW_STATE_DIR=/data/.openclaw` plus a Railway volume mount enforced via `requiredMountPath` in `railway.json`. |
| **Auth enforcement** | Refuses to boot if `OPENCLAW_GATEWAY_TOKEN` is missing, blank, contains whitespace, or is shorter than 24 chars. |
| **Reproducible builds** | `Dockerfile` pins the upstream image by SHA256 digest; daily cron repins. |
| **Healthchecks** | `railway.json` routes Railway's healthcheck to upstream's `/healthz`. |
| **Volume permissions** | Entrypoint runs as root, `chown`s `/data`, then drops to the non-root `node` user via `runuser`. |

## Deploy in three clicks

1. Click **Deploy on Railway** at the top.
2. Railway prompts for variables — all are **optional**. Either set them now or leave blank and add later from the service Variables tab.

     <tr><th>Recommended</th><th>Purpose</th></tr>

       <td><code>OPENCLAW_GATEWAY_TOKEN</code></td>

     </tr>
   </table>

   <table>

     <tr><td><code>GEMINI_API_KEY</code></td><td><a href="https://aistudio.google.com/app/apikey">aistudio.google.com</a></td></tr>
     <tr><td><code>OPENROUTER_API_KEY</code></td><td><a href="https://openrouter.ai/keys">openrouter.ai</a></td></tr>

3. Click **Deploy**. Railway provisions the service, attaches a 5 GB volume at `/data`, and waits for `/healthz` to go green.

Visit the service's public URL — authenticate with your gateway token to
configure channels and keys.

## Configuration reference

<details>
<summary><strong>Authentication (token policy)</strong></summary>

| Variable | Behaviour |
| --- | --- |
| `OPENCLAW_GATEWAY_TOKEN` (or `OPENCLAW_GATEWAY_PASSWORD`) | Bearer auth for the gateway. **Optional.** If set, the wrapper validates strength (≥24 chars, no whitespace) and rejects weak tokens before the gateway starts. If unset, upstream openclaw auto-generates a strong token on first boot and persists it under `OPENCLAW_STATE_DIR` (default `/data/.openclaw/gateway.json`). |

To retrieve an auto-generated token after first boot: open the Railway service → **Shell** (or use `railway run`) and `cat /data/.openclaw/gateway.json | jq .auth` (or grep for the token). For a more predictable workflow, set `OPENCLAW_GATEWAY_TOKEN` in service variables yourself before first deploy.

</details>

<details>
<summary><strong>Optional env vars set by the wrapper</strong></summary>

| Variable | Default | Purpose |
| --- | --- | --- |
| `OPENCLAW_STATE_DIR` | `/data/.openclaw` | Persistent state on the mounted volume. |
| `OPENCLAW_WORKSPACE_DIR` | `/data/workspace` | Persistent workspace files. |
| `OPENCLAW_REQUIRE_CONFIGURED` | unset | Set to `1` to drop the `--allow-unconfigured` flag — the gateway refuses to boot without LLM keys. |
| `PORT` | `8080` (Railway sets) | The port the gateway binds to. |

The gateway always binds to LAN (`--bind lan` = `0.0.0.0`); this is hard-coded
in the entrypoint because Railway's proxy can't reach a loopback bind. There
is no env var to override it — if you need loopback for some reason, fork the
wrapper.

</details>

<details>
<summary><strong>LLM provider, channel, and tool keys</strong></summary>

See [openclaw `.env.example`](https://github.com/openclaw/openclaw/blob/main/.env.example)
for the full list (LLM providers, channel tokens, tool keys like
`BRAVE_API_KEY`, `PERPLEXITY_API_KEY`, voice/media keys, OTel exporters, …).

By default `--allow-unconfigured` is passed so the gateway boots without keys
(configure via the gateway UI/API after deploy). Set
`OPENCLAW_REQUIRE_CONFIGURED=1` to refuse boot until keys are set.

</details>

## Maintainer setup

> One-time setup, after forking this repo to your own GitHub account.

1. **Push the fork** to GitHub. Confirm the `sync-upstream` and `ci` workflows enable themselves under the Actions tab.
2. **Deploy from GitHub once manually**:
   - Go to [railway.com/new](https://railway.com/new) → **Deploy from GitHub repo** → select this repo.
   - Wait for first deploy to succeed (`/healthz` goes green).
3. **Attach a volume**:
   - In service settings, attach a **5 GB volume** mounted at `/data`. Wait for the redeploy.
4. **Set required env vars on the service**:
   - `OPENCLAW_GATEWAY_TOKEN` — paste a value generated by `openssl rand -hex 32`. (You'll change this to `${{secret(32)}}` when you publish the template, so each deploy gets a fresh token.)
5. **Verify the service is healthy** — open the public URL; gateway responds.
6. **Generate a template**:
   - Project Settings → **Generate Template from Project**.
   - Mark `OPENCLAW_GATEWAY_TOKEN` as `${{secret(32)}}` (no spaces inside the braces — Railway's parser is strict).
   - Mark LLM provider keys as **optional** prompts with descriptions linking to provider signup pages.
   - Confirm the volume + mount path are captured in the template.
   - Publish the template.
7. **Update the Deploy button** in this README:
   - Replace the URL after `[![Deploy on Railway](https://railway.com/button.svg)]` with the URL Railway returned when you published — typically `https://railway.com/deploy/<slug>?referralCode=<your-code>` (Railway gives you the full URL on the template page).
   - Commit + push.

After step 7, anyone clicking Deploy spawns a fresh Railway project from
**your** repo, and inherits all daily auto-updates as they're committed.

> [!WARNING]
> **Branch protection caveat**: if you enable required-PR or required-review rules on `main`, the daily sync workflow fails at `git push` because the default `GITHUB_TOKEN` cannot bypass protection. Either exempt `github-actions[bot]` from rules, or rework the sync workflow to open a PR instead of pushing directly. The current setup assumes solo / unprotected `main`.

## How auto-update works

```
              ┌──────────────────────────────────────────────────────┐
              │  GitHub Actions: sync-upstream (daily 06:17 UTC)    │
              ├──────────────────────────────────────────────────────┤
              │  1. Read tag from UPSTREAM_TAG (default: latest)    │
              │  2. Resolve tag → digest via GHCR API (3× retry)    │
              │  3. Compare with current FROM in Dockerfile         │
              │  4. If different: pull image, smoke-test /healthz   │
              │  5. Rewrite Dockerfile FROM line (re.subn assert)   │
              │  6. git push                                         │
              │  7. On failure: open dedup'd "sync-failure" issue   │
              └──────────────────────┬───────────────────────────────┘
                                     │
                                     ▼
              ┌──────────────────────────────────────────────────────┐
              │  Railway GitHub integration                         │
              │  Sees push → rebuilds → pulls new image digest      │
              │  → restarts service. Templated services follow.     │
              └──────────────────────────────────────────────────────┘
```

> [!IMPORTANT]
> The smoke test in `sync-upstream.yml` validates the **upstream image directly**.
> The wrapper itself (Dockerfile, `start.sh`, env validation) is exercised by
> `ci.yml`'s build job on every push and PR. So a wrapper-side regression is
> caught at CI time; an upstream regression is caught here before the bump
> commits.

**To track a different upstream channel** — edit `UPSTREAM_TAG`:

| Value | Meaning |
| --- | --- |
| `latest` (default) | Most recent stable release. |
| `main` | Bleeding edge — every commit on upstream's main branch. |
| `2026.1.30` | Frozen at a specific version. No auto-bumps. |

After editing, push — the sync workflow picks up the change immediately and
repins.

## Files

| File | Purpose |
| --- | --- |
| [`Dockerfile`](Dockerfile) | Wraps `ghcr.io/openclaw/openclaw@<digest>`; root user fixes volume perms, drops via `runuser`. |
| [`start.sh`](start.sh) | Validates env, ensures state dirs, drops privileges, execs gateway with `--bind lan --port $PORT --allow-unconfigured`. |
| [`railway.json`](railway.json) | DOCKERFILE builder, `/healthz` healthcheck, `restartPolicyType: ALWAYS`, `requiredMountPath: /data`. |
| [`UPSTREAM_TAG`](UPSTREAM_TAG) | Single line: which upstream tag to track. |
| [`.github/workflows/sync-upstream.yml`](.github/workflows/sync-upstream.yml) | Daily cron: resolve → smoke-test → repin → push. Issue on failure. |
| [`.github/workflows/ci.yml`](.github/workflows/ci.yml) | Shellcheck + JSON validation + image build + wrapper smoke-test. |
| [`scripts/resolve-upstream-digest.sh`](scripts/resolve-upstream-digest.sh) | Resolves a tag to its current GHCR digest (3× retry, validates token format). |
| [`scripts/smoke-test.sh`](scripts/smoke-test.sh) | Boots a container, polls `/healthz`, redacts test token from logs on failure. |
| [`NOTICE`](NOTICE) | Upstream MIT attribution. |

## Local development

```sh
# Build the wrapper image
docker build -t openclaw-railway:dev .

# Run it (token is required — even locally — by start.sh)
docker run --rm -it \
  -e OPENCLAW_GATEWAY_TOKEN="$(openssl rand -hex 32)" \
  -p 8080:8080 \
  openclaw-railway:dev

# In another terminal
curl http://localhost:8080/healthz
```

To dry-run what the daily sync workflow does:

```sh
bash scripts/resolve-upstream-digest.sh latest
bash scripts/smoke-test.sh "ghcr.io/openclaw/openclaw@$(bash scripts/resolve-upstream-digest.sh latest)"
```

## Cost

Railway pricing changes; check [docs.railway.com/reference/pricing](https://docs.railway.com/reference/pricing) for current numbers. Practical guidance:

- **Trial / Free tiers** (0.5 GB RAM, 0.5 GB volume) are insufficient — openclaw needs ≥2 GB RAM and the volume fills fast.
- **Hobby ($5/mo + $5 credit)** is the realistic floor: 5 GB volume, comfortable RAM ceiling.
